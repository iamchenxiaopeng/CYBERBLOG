## 前言：为什么写这篇

我想做一个个人博客——赛博朋克风格，前后端分离，能写 MD 文章、有点赞和评论功能。

听起来是个简单的需求对吧？但当我把这个想法扔给 AI 的时候，事情远没有想象中顺利。

这篇文章记录了这个项目从零到上线的完整过程，以及我作为一个**普通人**（不是 AI，不是代码生成器）在这个过程中的真实感受。

> **核心问题：AI 写代码很快，但 AI 不会替你想"完整的需求是什么"，更不会在你不说的时候主动问。**

## 第一阶段：需求描述的艺术

### 我的第一次尝试

我本来想说："帮我做个博客。"

但我想了想，如果我这么说，AI 大概率会给我一个最简单的单页 HTML，或者一个 Markdown 转 HTML 的静态站点生成器——技术上能用，但和我心里想的差了十条街。

### 我实际说的

```
我想开发一个赛博朋克风格的个人博客。
前端用 Vue3，后端用 Java Spring Boot，前后端分离。
功能：点赞、评论、注册/登录（一键注册）、MD 文档上传与渲染、手写博客。
```

你注意到区别了吗？我说清楚了：

- **风格**（赛博朋克）——不是"好看一点"这种模糊词
- **技术栈**（Vue3 + Java）——不是我猜你会选的技术
- **功能列表**（具体到每一项）——不是"该有的都有"
- **交互细节**（一键注册）——不是"注册功能"

### AI 一次就做对了什么

- 技术选型合理：Vue3 + Vite + Pinia + Spring Boot 3 + MyBatis-Plus
- CSS 主题惊艳：扫描线、Glitch 动效、霓虹色、Orbitron 字体
- API 设计规范：RESTful 风格、JWT 鉴权、统一响应格式
- 数据库设计合理：用户、文章、评论、点赞四张表

### 🎯 经验一：需求描述是"翻译"，不是"交代"

如果你想让 AI 写出你想要的代码，你需要像翻译一样把你的想法**转换**成 AI 能理解的精确描述。关键词越是模糊（"好看"、"好用"、"快"），AI 的输出就越是随机。

| 你说的话 | AI 的理解 | 你实际想表达 |
|---------|----------|------------|
| "做个好看的博客" | 随便选一套 CSS 主题 | 赛博朋克风格，霓虹色+扫描线 |
| "要有登录功能" | 用户名+密码登录 | 一键注册，不需要邮箱验证 |
| "能写文章" | 一个 textarea | MD 编辑器 + 实时预览 + 文件上传 |

**教训：把需求当作给外包团队的 PRD 来写，越具体越好。**

## 第二阶段：点赞功能的两次翻车

这是整个项目中最典型的「AI 写了代码但用户活受罪」的案例。

### 翻车一号：点完赞，什么都没发生

我兴冲冲打开博客，点了一下文章下面的赞按钮。

**无事发生。**

不是后端报错了，不是网络断了——是前端代码在未登录时**静默 return 了**，连个提示都没有。

```javascript
// AI 写的代码（简化版）
async function toggleLike(articleId) {
  if (!isLoggedIn.value) return;  // ← 这行是罪魁祸首
  // ... 后续逻辑
}
```

作为一个用户，我点击了一个按钮，它看起来是可点击的（hover 有高亮），我点了——然后什么都没发生。没有提示，没有弹窗，没有反馈。我最开始甚至以为是按钮坏了。

### 为什么会出这种问题？

因为 AI 在写代码的时候，**默认用户一定知道"未登录不能点赞"这个规则**。AI 没有"站在用户角度思考"的能力——它只关心逻辑正确性，不关心用户体验。

### 翻车二号：未登录就不能点赞？

修好了提示之后，我又想：为什么未登录就不能点赞？很多博客网站（比如 Medium）不登录也能点赞，只是用 Cookie 或浏览器指纹来标识。

于是我跟 AI 说：

```
未登录也可以点赞，点赞用户记录为浏览器对象的电脑用户名。
```

AI 理解了我的意图，做了以下改造：

1. 数据库 `likes` 表新增 `guest_id` 字段（VARCHAR 64）
2. 后端 `Like` 实体、Service、Controller 全部支持两种模式（`userId` 或 `guestId`）
3. 前端生成并存储浏览器 UUID，未登录时自动带 `X-Guest-Id` 请求头
4. 唯一索引拆分为两个：`uk_like_user` 和 `uk_like_guest`

### 🎯 经验二：告诉 AI 边界条件，而不只是核心逻辑

AI 擅长写"快乐路径"——用户已登录、网络正常、数据合法。但真实世界全是边界情况：

| 场景 | AI 默认行为 | 你该告诉 AI |
|------|------------|------------|
| 未登录点赞 | 静默跳过 | 弹出提示告知用户 |
| 未登录用户 | 不能点赞 | 用浏览器 UUID 标识 |
| 空文章标题 | 报错或不处理 | 前端表单校验 |
| 恶意超大文件 | 不限制 | 文件大小上限 |
| 并发点赞 | 可能出现重复 | 加唯一索引防重 |

**让 AI 帮你做一次"边界条件审查"：在写完主要功能后，追加一句"检查一下这个功能有哪些边界情况没处理"。**

## 第三阶段：容器化与一键部署

### 启动项目到底有多麻烦？

写完代码后，我想跑起来看看效果。然后发现启动这个项目需要：

1. 启动 MySQL 数据库
2. 初始化数据库和表
3. 启动 Redis
4. 启动 MinIO 对象存储
5. 用 Maven 编译 Spring Boot 项目
6. 启动后端（配置好数据库连接等）
7. 用 npm 安装前端依赖
8. 启动 Vite 开发服务器

**即使每一步都顺利，手动操作也要 10 分钟。** 而且每次重启电脑后都要重新来一遍。

### 我的要求：一行命令启动

我跟 AI 说：

```
这个项目运行太麻烦了，让它支持一行命令启动。
```

AI 做了这些事情：

1. **给后端写 Dockerfile**：多阶段构建（maven:3.9 编译 → eclipse-temurin:17-jre-alpine 运行）
2. **给前端写 Dockerfile**：多阶段构建（node:20 构建 → nginx:alpine 托管）
3. **写 nginx.conf**：SPA fallback + `/api/` 反代到后端
4. **重写 docker-compose.yml**：5 个服务（mysql/redis/minio/backend/frontend），带 healthcheck 依赖链
5. **写 start.bat（Windows）和 start.sh（Linux/Mac）**

结果：**`.\start.bat` 或 `bash start.sh`，一行搞定。** 首次启动 3-5 分钟（编译+构建），之后秒启。

### 🎯 经验三：把"运维的苦"也翻译成需求

很多开发者只告诉 AI "写个 Dockerfile"，但没告诉它：

- 服务之间的启动顺序（MySQL 先启动，后端再启动，前端最后）
- 健康检查机制（怎么判断一个服务"准备好了"）
- 环境变量的注入方式
- 一键脚本需要支持哪些子命令（start/stop/restart/logs）

**关键句式："让这个项目支持一行命令启动，包括数据库、缓存、存储、前后端所有服务。"** 越完整的语句，AI 越不会遗漏。

## 第四阶段：Windows 用户的噩梦三连

这是最让我崩溃的环节。代码写完了，Docker 配置写完了，一行启动命令也写好了。我兴冲冲运行 `start.bat`——

### 噩梦一：中文乱码

```
'鎵ц' 不是内部或外部命令，也不是可运行的程序
'鍚姩鎴愬姛锛' 不是内部或外部命令...
```

`start.bat` 里的中文提示被 Windows CMD（默认 GBK 编码）当成了命令来执行！

**根因**：Windows CMD 默认用 GBK 编码读 .bat 文件，而 AI 写的中文是 UTF-8 编码。字符错位后，Windows 把乱码当成命令执行了。

**修复**：
1. 在 `start.bat` 开头加 `chcp 65001 >nul` 切换到 UTF-8
2. 把所有中文提示改成英文（更彻底）

**如果你让 AI 一开始就用英文写脚本提示，就不会有这个问题。**

### 噩梦二：TypeScript 类型推断错误

乱码修好后，Docker 开始构建前端。然后：

```
ERROR TS2322: Type 'string' is not assignable to type '"write" | "upload"'.
```

AI 写了一个 `modes` 数组：

```typescript
const modes = [
  { key: 'write', label: '写作' },
  { key: 'upload', label: '上传' },
];
```

TypeScript 把 `key` 推断成了 `string` 类型。但当这个值被传给一个要求 `'write' | 'upload'` 联合类型的地方时，编译器报错了。

**修复**：给数组加显式类型注解：

```typescript
const modes: { key: 'write' | 'upload'; label: string }[] = [
  { key: 'write', label: '写作' },
  { key: 'upload', label: '上传' },
];
```

**这是 AI 写 TypeScript 时的经典疏忽**：字面量类型没有显式声明，被 TS 推断为更宽泛的类型。

### 噩梦三：后端端口随机漂移

Docker 构建成功了，但我发现后端连不上。一查日志：**Spring Boot 启动在了 8081 端口，而不是配置的 8080。**

原因是 Docker 容器里有一个被终止的后台进程占着 8080 端口，Spring Boot 的"智能"机制自动换了一个端口。

**修复**：在启动参数里强制指定 `--server.port=8080`，不允许自动更换。

### 🎯 经验四：把错误日志原封不动贴给 AI

三次翻车的共同点：**我做的第一件事就是把完整的错误日志贴给 AI。**

不要自己尝试理解、翻译或总结错误信息。AI 对错误日志的理解比你好得多——它见过几百倍的同类错误。你只需要把错误日志原样贴过去，AI 几乎每次都能秒出修复方案。

**关键动作：**
- ✅ 把完整错误栈贴给 AI
- ✅ 附上相关代码文件路径
- ❌ 不要自己"总结"错误（"好像是什么编码问题"）
- ❌ 不要只贴最后一行错误信息

## 第五阶段：文档沉淀与分享

当项目能顺利跑起来之后，我让 AI 做了一件很重要的事：**写自动化原理文档**。

```
详细讲讲你都做了什么让项目实现了自动化，并生成一个说明文档。
```

这份文档（`AUTOMATION.md`）涵盖了：

- 多阶段 Docker 构建的原理和优势
- docker-compose 服务编排策略
- healthcheck 依赖链的设计思路
- 环境变量注入机制
- nginx 反向代理配置
- 一键脚本的工作原理

> **知道"怎么用"很重要，但知道"为什么这样设计"更重要。** 前者能让你运行项目，后者能让你**在其他项目中复刻这个模式**。

### 🎯 经验五：让 AI 解释它自己的设计决策

每完成一个大功能，追加一句："解释一下你是怎么做这个的，以及为什么选择这样做。" 这会帮你建立对代码的**深度理解**，而不仅仅是"能跑就行"。

## 总结：人 + AI 协作的六条铁律

经过这个项目的完整开发流程，我总结了六条和 AI 高效协作的铁律：

### 1. 需求像 PRD，不像聊天

把需求当作给外包团队的产品需求文档来写。风格、技术栈、功能列表、交互细节——越具体，AI 越不会跑偏。

**坏：** "做个博客"
**好：** "用 Vue3 + Spring Boot 做赛博朋克风格博客，包含注册登录（一键注册）、MD 文章上传、点赞、评论功能"

### 2. 主动触发边界检查

AI 默认写"快乐路径"。写完每个功能后，追问一句："检查一下有什么边界情况没处理？"

**触发词：** "检查边界情况"、"如果未登录会怎样"、"空数据会报错吗"、"并发安全吗"

### 3. 运维需求要完整

启动项目不只是"docker-compose up"那么简单。完整的需求包括：

- 所有服务的启动顺序
- 健康检查机制
- 环境变量定义
- 一键脚本（含 stop/restart/logs 子命令）

**关键句式：** "让这个项目支持一行命令启动，包括数据库、缓存、存储、前后端所有服务。"

### 4. 错误日志原样贴

不要翻译、不要总结、不要只贴最后一行。完整的错误栈 + 相关文件路径 = AI 秒出修复方案。

### 5. 让 AI 解释它的设计

每完成一个模块，让 AI 解释它是怎么做的以及为什么选择这样做。这不是多余的步骤——这是你从"会用 AI"到"会和 AI 一起成长"的分水岭。

**触发词：** "解释一下你的设计思路"、"为什么选择这个方案而不是 XX"

### 6. 小步迭代，不要一步到位

这个项目不是一次写完的，而是：

```
基本框架 → 示例文章 → 修复点赞 → 匿名点赞 → 容器化 → 修复乱码 → 修复 TS 错误 → 修复端口 → 文档沉淀
```

每一步只做一件事，每一步都立即验证结果。**如果一次让 AI 做太多事，出问题时你不知道该从哪里开始排查。**

## 后记

从"想做"到"上线"，这个博客项目经历了大约 9 个迭代回合。AI 写了 95% 的代码，但关键的 5%——决定做什么、发现哪里不对、告诉 AI 该怎么修——全部来自人的判断。

**AI 是一个能写代码的极客朋友，但它需要你来告诉它：** 做什么、为什么做、什么才是"对的"。

希望这篇记录能帮你更高效地和 AI 合作。毕竟，工具的好坏不取决于工具本身，而取决于**你怎么用它**。
